MIT6.S081-Lab2 syscall: System calls

实验目的

In the last lab you used systems calls to write a few utilities. In this lab you will add some new system calls to xv6, which will help you understand how they work and will expose you to some of the internals of the xv6 kernel.

本实验主要是为xv6增加两个系统调用，通过实现系统调用来帮助理解系统调用的执行流程，系统调用的具体执行在kernel中，借此可以进一步了解xv6的kernel实现。

首先简单介绍下xv6中的系统调用。

kernel中：

  • 所有系统调用都对应一个编号，定义在kernel/syscall.h中。

  • 所有系统调用都是通过void syscall(void);函数来调用的，此函数从进程的trapframe(保存进程从用户态进入内核态时的寄存器等值，进程调用系统调用会从用户态进入到内核态。)中获取系统调用的编号，然后索引到系统调用的函数指针，执行对应的系统调用(详见kernel/syscall.c)。

  • 所有系统调用函数都是uint64 sys_name(void);格式的，系统调用的参数从进程的trapframe中获取(即trapframe中保存的寄存器值)，系统调用的返回值为-1对应错误。

    kernel/syscall.c 中封装了几个函数argint、argaddr、argstr来获取系统调用参数。

用户态：

  • user/user.h 中声明了给用户封装好的的C语言类型的系统调用函数。
  • user/usys.pl中为每个系统调用函数(封装好的)生成一段汇编代码(user/usys.S)，汇编代码主要完成保存系统调用编号到指定寄存器(li: load immediate)，调用函数syscall执行系统调用(ecall: 进入内核态调用syscall)，从内核态返回到用户态(ret)。
  • 用户调用系统调用即通过user/user.h 中的C语言类型函数调用。

System call tracing

In this assignment you will add a system call tracing feature that may help you when debugging later labs. You'll create a new trace system call that will control tracing. It should take one argument, an integer "mask", whose bits specify which system calls to trace. For example, to trace the fork system call, a program calls trace(1 << SYS_fork), where SYS_fork is a syscall number from kernel/syscall.h. You have to modify the xv6 kernel to print out a line when each system call is about to return, if the system call's number is set in the mask. The line should contain the process id, the name of the system call and the return value; you don't need to print the system call arguments. The trace system call should enable tracing for the process that calls it and any children that it subsequently forks, but should not affect other processes.

Sysinfo

In this assignment you will add a system call, sysinfo, that collects information about the running system. The system call takes one argument: a pointer to a struct sysinfo (see kernel/sysinfo.h). The kernel should fill out the fields of this struct: the freemem field should be set to the number of bytes of free memory, and the nproc field should be set to the number of processes whose state is not UNUSED. We provide a test program sysinfotest; you pass this assignment if it prints "sysinfotest: OK".

sysinfo函数接收一个struct sysinfo* 类型的参数，sysinfo函数的作用就是收集系统的信息，将其放在参数所指的地址中。sysinfo函数调用发生在用户态，其函数参数是一个用户地址空间的地址，而其底层的sys_sysinfo系统调用执行时发生在kernel中，因此收集的信息不能直接赋值，需要借用kernel/vm.c中的函数copyout将kernel中收集的信息copy到参数对应的用户地址空间中。

sys_sysinfo系统调用主要分为两步：

  1. 收集信息

    • 收集空闲内存数量(byte)

      按照hints, 在kernel/kalloc.c中增加一个函数uint64 get_freemem()

      kernel/kalloc.c:

      ​ 系统的空闲内存以链表的形式保存在kmem.freelist中，链表中每个节点对应一页，节点的值是一页的起始地址。

      ​ 系统的内存分配和内存回收均以页为单位，每页PGSIZE个byte，分配时从链表头取下一页，回收时向链表头插入一页。

      ​ kem.freelist在系统启动时通过kernel/kalloc.c中的kinit()函数初始化。

    • 收集未使用进程数量

      按照hints, 在kernel/proc.c中增加一个函数uint64 get_nproc()

      kernel/proc.c:

      ​ 系统中初始时设置了NPROC个进程，保存在进程数组struct proc proc[NPROC]中。遍历进程数组，统计UNUSED的进程数量即可。

  2. 拷贝至用户空间

    addr是系统调用的参数，对应用户态传入的用户地址空间地址，由此+进程的页表可定位至相应的物理地址。

    (char*)(&info)是kernel中的地址，由此+内核页表可定位至对应的物理地址。

    copyout 实际上就是将kernel中一块地址对应物理地址上的内容拷贝至用户虚拟地址空间中一块地址对应的物理地址中。只不过copyout 调用时在kernel，此时paging hardware中使用的就是内核页表，因此内核页表不用显示给出。
