MIT6.S081-Lab3 pgtbl: Page tables

实验目的

In this lab you will explore page tables and modify them to simplify the functions that copy data from user space to kernel space.

本实验主要是打印页表，为每个进程创建一个内核页表以及借助内核页表来简化向内核拷贝数据的操作。通过本次实验，对页表机制、内核虚拟地址空间、用户虚拟地址空间以及进程切入切出内核流程的理解将大大加深。

前置知识

下文陈述的前置知识是以xv6系统的设计为基础的，真实的操作系统中一定是更加复杂和多样的，不过基本的原理都差不多。

页表机制

  • 无论是在用户虚拟地址空间中还是在内核虚拟地址空间中，所有指令用到的地址都是虚拟地址。

  • 内核页表全局唯一，用于将在内核态中执行的指令的虚拟地址转换为物理地址。

    用户页表每个进程一个，用于将进程态执行的指令的虚拟地址转换为物理地址。

  • 虚拟地址向物理地址的转化是通过硬件实现的，硬件通过查找页表将虚拟地址转换为物理地址。

xv6中的页表结构如下图:

虚拟地址占用8个字节，共64位，xv6运行在Sv39 RISC-V上，因此虚拟地址中只使用64位中的低39位(高25位可以视作全0)。

物理地址占用8个字节，共64位，xv6中，物理地址只使用64位中的低56位(高8位可以视作全0)。

系统的内存分配和回收以页为单位，一页的大小是4096byte, 页内地址用12位表示。

虚拟地址(39位)中高27位为虚拟页号，用于索引页表项，低12位为页内地址。

物理地址(56位)中高44位为物理页号，低12位为页内地址。

页表项占用8个字节，共64位，每一页可以包含512个页表项。页表项只使用64位中的低54位（高10位为保留位）。页表项(54位)的高44位是物理页号，低10位每个位代表一个标志位，标识页表项的一些状态和特征。

硬件通过虚拟地址(39位)中的虚拟页号(27位)去页表中查找对应的页表项(54位)，由页表项中的物理页号(44位)拼接虚拟地址中的页内地址(12位)得到物理地址(56位)。

页表是三级结构的，第一级页表所占的一页内存的首地址(物理地址)保存在satp寄存器中，每个CPU核心都有一个satp寄存器。

第一级页表只占用一页内存，包含512个页表项，每个页表项对应第二级页表的一页。

第二级页表占用512页，包含512*512个页表项，每个页表项对应第三级页表的一页。

第三级页表占用512*512页，包含512*512*512个页表项，每个页表项对应一个虚拟地址。

虚拟地址所要索引的页表项是第三级页表中的页表项。

页表结构图如下：

虚拟页号为27位，由虚拟页号去页表中查找物理页号(44位)的过程如下：

  1. 由高9位去satp寄存器中保存的物理地址处索引一个页表项(54位)，此页表项的物理页号(高44位)拼接全0的12位得到56位物理地址(第二级页表的对应页的首地址)。
  2. 由中间9位去1中得到的56位物理地址处索引一个页表项(54位)，此页表项的物理页号(高44位)拼接全0的12位得到56位物理地址(第三级页表的对应页的首地址)。
  3. 由低9位去2中得到的56位物理地址处索引一个页表项(54位)，得此页表项的物理页号(高44位)。

由3中得到的物理页号(44位)拼接虚拟地址中的页内地址(12位)即为该虚拟地址对应的物理地址(56位)。

内核虚拟地址空间

左边为内核的虚拟地址空间，虚拟地址编址从0开始，到MAXVA结束(2^39-1)。

右边为物理内存的地址空间，物理地址编址从0开始，到2^56-1结束。

内核虚拟地址与物理地址的映射保存在内核页表中，全局唯一，内核虚拟地址的0-PHYSTOP与物理地址直接映射。内核虚拟地址的0-KERNBASE地址区间映射的是一些设备地址，KERNBASE-PHYSTOP映射的是物理内存地址(即RAM)。

trampoline页(保存的是发生trap时执行的代码)在内核页表中映射在内核虚拟地址的最高处(MAXVA - PGSIZE)，在进程页表中也是映射在进程虚拟地址空间的最高处(MAXVA - PGSIZE)。

trampoline页(物理页面)在内核虚拟地址空间和进程虚拟地址空间中映射到同一处的好处是：

切换页表时（写satp寄存器), trampoline代码仍然能够正确执行，由于陷入trap时(执行trampoline页面的代码)，往往涉及进程空间到内核空间的转换，映射到同一处就很方便。

trampoline页面在内核页表中被映射两次也有这方面原因，一次映射是地址范围0-PHYSTOP的直接映射，另一次映射在最高处(与进程页表统一)。

Kstack0..对应进程的内核栈，系统初始化时为每个进程分配了一个内核栈，用于进程在内核中执行代码。每个内核栈附加一个保护页，保护页仅仅是一个虚拟地址页面，没有物理内存中的物理页面，内核页表中也没有其内核虚拟地址对应的页表项。因此当进程在内核栈中运行代码超过自己的内核栈地址进入保护页时，会触发pagefault，而不是进入其他内核栈的内核地址区间，起到保护作用。

用户虚拟地址空间

用户页表中把trampoline物理页面映射在用户虚拟地址空间的最高处(MAXVA-PGSIZE)，和内核页表的做法一样。trapframe物理页面是每个进程一个，在进程初始化时分配，其作用是保存进程陷入trap时的寄存器值，用于进程陷入trap后返回时恢复现场。

A kernel page table per process

Your first job is to modify the kernel so that every process uses its own copy of the kernel page table when executing in the kernel. Modify struct proc to maintain a kernel page table for each process, and modify the scheduler to switch kernel page tables when switching processes. For this step, each per-process kernel page table should be identical to the existing global kernel page table. You pass this part of the lab if usertests runs correctly.

原始的xv6中内核页表全局唯一，用户页表每个进程一个，进程的数据页，代码页被映射在用户页表中，当进程因系统调用或其他原因进入内核时，CPU切换到内核页表(改写satp寄存器)。内核页表中没有进程那些数据、代码页的映射，因此用户虚拟地址空间中的虚拟地址就不能直接使用了，需要手动借助用户页表去找到对应的物理地址。

本实验的目的就是为每个进程增加一个内核页表，此内核页表在复制原有内核页表的基础上，加了进程数据、代码页的映射，当进程进入内核时，切换到进程的内核页表。此时即使在内核虚拟地址空间中，因为进程内核页表中包含了用户虚拟地址空间的映射，用户虚拟地址空间的地址可以直接使用，不再需要通过手动通过进程的用户页表去找对应的物理地址。

实验A kernel page table per process中我们给每个进程创建了一个内核页表，进程的内核页表是在初始化进程时创建的，进程内核页表的页表项初始时只有全局内核页表的页表项+进程的内核栈对应的页表项，而没有进程的数据、代码、进程用户栈等的页表项。要想达到进程在内核空间中也能直接使用用户空间的虚拟地址，需要把用户空间的页表项拷贝至进程内核页表中。

以copyin函数为例，copyin函数一般是进程在内核空间中调用，作用是从用户地址空间中拷贝数据至内核地址空间中。

之前的做法：

  1. 查找pagetable将进程虚拟地址srcva转为物理地址。

  2. 内核虚拟地址空间中内存范围内虚拟地址与物理地址直接映射。1中srcva转成的物理地址在内核中可以当作内核虚拟地址使用。

  3. copyin拷贝的过程是从srcva转成的物理地址(等同于内核虚拟地址)拷贝至dst(内核虚拟地址)。

  4. copyin是在内核中执行，内核中运行的代码使用的是全局内核页表，3中的虚拟地址通过内核页表转化为物理地址(这里的转化是硬件实现的)，真正拷贝是从一个物理地址到另一个物理地址。

    可能有点绕，但必须明确一点内核中使用的也是虚拟地址，真正的拷贝是发生在物理地址之间的，巧妙的地方在于：内核虚拟地址空间中内存范围内虚拟地址与物理地址直接映射。

现在想要的效果：

copyin在内核空间中调用，srcva这个用户虚拟地址现在在进程内核页表中也有对应的映射，因此可以直接调用memmove，从内核的一个虚拟地址拷贝数据到内核的另一个虚拟地址，前提在于进程的内核页表中有这两个虚拟地址的映射。
