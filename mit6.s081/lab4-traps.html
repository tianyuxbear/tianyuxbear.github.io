MIT6.S081-Lab4 traps: Trap

实验目的

This lab explores how system calls are implemented using traps. You will first do a warm-up exercises with stacks and then you will implement an example of user-level trap handling.

本实验通过对RISC-V汇编的寄存器等内容的介绍，对trampoline.S以及trap相关源码的阅读，深入了解了xv6的trap机制。RISC-V assembly实验强化了对RISC-V汇编知识的理解，Backtrace的实验会让自己对栈的使用理解更深刻，Alarm实验通过对中断的控制实现alarm效果，实验深入trap的源码,十分有助于对trap机制的理解。

前置知识

RISC-V汇编

详细内容建议仔细阅读：https://pdos.csail.mit.edu/6.828/2020/readings/riscv-calling.pdf

RISC-V的寄存器：

通常谈到寄存器时，我们一般使用寄存器的ABI Name,更加清晰明了，写汇编代码时也是用的ABI Name。寄存器的宽度为64位。

  • a0-a7：用于保存函数的整型参数。其中a0-a1还用于保存函数的整型返回值。函数返回值<=64位时，返回值保存在寄存器a0中，函数返回值>64位时，返回值保存寄存器a0和a1中。

    函数整型参数个数大于8个时，从第9个整型参数开始，函数整型参数保存在内存中。

  • fa0-fa7:用于保存函数的浮点型参数，其中a0-a1还用于保存函数的浮点型返回值。函数返回值<=64位时，返回值保存在寄存器fa0中，函数返回值>64位时，返回值保存寄存器fa0和fa1中。

    函数浮点型参数个数大于8个时，从第9个浮点型参数开始，函数浮点型参数保存在内存中。

Caller Saved(又称Callee-owned)寄存器和Callee Saved(又称Caller-owned)寄存器的区别：

  • Caller Saved Registers: 在函数调用过程(被调函数的执行过程)中不会被保存的寄存器。

    对于调用方，在调用函数前，已将Caller Saved 寄存器中的值(不是所有，根据需要保存)保存在自己的栈帧中，等函数调用返回时，调用方恢复Caller Saved寄存器中的值。(这部分保存与恢复工作是调用方自己完成的)。

    对于被调用方，它可以直接使用Caller Saved寄存器中的值并且覆盖，因为它知道调用方已经保存了Caller Saved寄存器中的值。

  • Callee Saved Registers: 在函数调用过程(被调函数的执行过程)中会被保存的寄存器。

    对于调用方，它不需要提前保存Callee Saved寄存器中的值，因为它知道被调用方在执行过程中会保存并在返回时恢复Callee Saved寄存器中的值。

    对于被调用方，如果需要使用Callee Saved寄存器中的值，必须先将Callee Saved寄存器中的值保存下来才能使用(保存在自己的栈帧上)，并且在返回调用方之前，将Callee Saved寄存器中的值恢复。(这部分保存与恢复工作是由被调用方完成的)。

Stack结构

stack的简易结构图:

学C语言时，老师常说函数运行在栈空间中。

怎么理解这句话呢？

首先假设我们写了个C程序，显而易见C程序中有个main函数，它是程序的入口。main函数中可能调用funcA、funcB、···，funcA中又可能调用funcAA...

无论函数怎么嵌套，请记住: 函数对应一个代码块，函数的调用对应一串代码的执行。函数的嵌套无非就是代码块的执行顺序，流程之类的。

栈空间中，每个函数对应一个栈帧(stack frame)。

回到C程序，C程序的执行对应一个进程，进程中有个栈空间，进程中第一个执行的函数是main函数，main函数的栈帧压在栈底，main函数调用funcA，funcA的栈帧压入栈，funcA调用结束，funcA的栈帧被弹出栈，函数的嵌套调用无非就是对应栈帧的压入和弹出，当main函数结束时，main的栈帧弹出栈空间，进程结束。

回到栈的简易结构图，栈空间是从高地址开始向低地址使用(可以把栈空间的最高地址看作栈底)，当我们想要创建一个新的Stack Frame(调用一个函数)的时候，总是对当前的Stack Pointer(可以理解为栈顶指针)做减法。

让我们看看Stack Frame中有哪些内容：

  • 保存的Callee寄存器内容。

    SP(Stack Pointer)寄存器：指向当前Stack Frame的低地址处(栈帧的最下方)。

    FP(Frame Pointer)寄存器: 指向当前Stack Frame的高地址处(栈帧的最上方)。

    联系上文，假设在main函数中调用funcA，funcA刚开始执行时，编译器将main函数的的栈帧的SP和FP寄存器值保存在funcA的栈帧中；当funcA执行过程中，SP和FP寄存器值为funcA函数栈帧的最下方地址和最上方地址；当funcA执行完毕返回main函数时，编译器将SP和FP寄存器的值恢复为原main函数栈帧的最下方地址和最上方地址。

  • 函数本地的定义的变量。

  • 函数的参数(参数>8个时从第9个参数开始，函数参数保存在栈帧中)。

  • Return address：当前函数的返回地址, 总是保存在当前Stack Frame的最上方。

    RA(Return address)寄存器：Caller Saved寄存器。

    联系上文，假设在main函数中调用funcA，调用funcA之前，main函数已经将RA寄存器的值保存在当前栈帧的最上方; funcA执行时，RA寄存器的被重写为funcA函数的返回地址，同时funcA栈帧的最上方也写入funcA函数的返回地址(RA寄存器值)。

    由此，Caller Saved 寄存器可以看作函数调用过程中其值保存在当前函数的栈帧中的寄存器。Callee Saved寄存器可以看作函数调用过程中其值保存在被调用函数的栈帧中的寄存器。

  • 指向前一个Stack Frame的指针也会保存在当前Stack Frame的固定位置。

    对于xv6,保存在Return address下面。

提个问题：为什么栈空间要从上到下增长(从高地址向低地址拓展)?

以APUE中经典内存分布的图为例：

假设栈空间从低地址向高地址扩展：

栈的起始地址是堆的终止地址，栈的终止地址是固定的。

由于栈的终止地址是固定的，一旦栈的大小需要动态调整，务必要更改栈的起始地址，那么整个栈空间的数据就需要拷贝移动。

另外，这种情况下栈的起始地址怎么设定？设为堆起始地址和栈终止地址的中点吗?(堆栈大小一致。)

程序不一定使用大致相等的栈空间和堆空间，有的程序可能使用堆空间更多，有的可能使用栈空间更多。

堆栈大小一致(平均分配)，对于使用堆空间较多的程序，可能导致堆空间溢出而栈空间大量闲置，对于使用栈空间较多的程序，可能导致栈空间溢出而堆空间大量闲置。

所以栈从高地址向低地址扩展：

  1. 栈起始地址固定，栈大小动态调整时无需移动栈中数据。
  2. 堆向上增长，栈向下增长，可以最大程度利用剩余的地址空间，适应各种程序不同的堆栈内存需求。

Trap机制

what is trap？

trap: 用户空间<==>内核空间之间的转换。

用户空间：user权限、进程虚拟地址空间 内核空间：supervisor权限、内核虚拟地址空间

trap往往发生在以下三种情况：

  • 程序执行系统调用。(ecall)

  • 程序出现了类似page fault、运算时除以0的错误。(异常)

  • 设备中断(输入输出等 device interrupt)、定时器中断(timer interrupt)。(中断)

中断和异常的区别：

前置知识中提到32个整型寄存器和32个浮点型寄存器，一共64个64位宽度的寄存器。

程序执行过程中主要使用这64个普通寄存器 以及 pc(Program Counter Register)程序计数器寄存器。

下面英文解释来自xv6文档原文，翻译成中文总感觉差点意思。

除此之外，还有一些特殊的控制寄存器：

  • 标识当前mode的寄存器：表明当前是处于supervisor mode or user mode.

  • satp(Supervisor Address Translation and Protection): 保存指向page table的物理内存地址(内核空间中指向内核页表，用户空间中指向进程页表)。

  • stvec(Supervisor Trap Vector Base Address Register): The kernel writes the address of its trap handler here; the RISC-V jumps here to handle a trap.

  • sepc(Supervisor Exception Program Counter): When a trap occurs, RISC-V saves the program counter here (since the pc is then overwritten with stvec). The sret (return from trap) instruction copies sepc to the pc. The kernel can write to sepc to control where sret goes.

  • sscratch(Supervisor Scratch Register):The kernel places a value here that comes in handy at the very start of a trap handler.(从源码中看，这个寄存器其实就是一个用于暂存的寄存器。)

  • scause:The RISC-V puts a number here that describes the reason for the trap.

  • sstatus： The SIE bit in sstatus controls whether device interrupts are enabled. If the kernel clears SIE, the RISC-V will defer device interrupts until the kernel sets SIE. The SPP bit indicates whether a trap came from user mode or supervisor mode, and controls to what mode sret returns.

CPU hardware’s trap handling sequence for all trap types：

  1. If the trap is a device interrupt, and the sstatus SIE bit is clear, don’t do any of the following.
  2. Disable interrupts by clearing SIE.
  3. Copy the pc to sepc.
  4. Save the current mode (user or supervisor) in the SPP bit in sstatus.
  5. Set scause to reflect the trap’s cause.
  6. Set the mode to supervisor.
  7. Copy stvec to the pc.
  8. Start executing at the new pc.

三种trap类型之间的共性表明内核可以用一个单一的代码路径来处理所有的trap。

但事实上，根据具体情况，有以下两种trap handler(对应于两个不同的stvec值)：

  • traps from user space
  • traps from kernel space

Note that the CPU doesn’t switch to the kernel page table, doesn’t switch to a stack in the kernel, and doesn’t save any registers other than the pc. Kernel software must perform these tasks.

trap处理流程中切换内核页表(write satp)，切换到内核栈(write sp), 保存64(32 + 32， xv6中好像只保存了31个整型(0号恒0不保存)的)个普通寄存器值，这些工作不是硬件完成的，是软件(汇编代码)完成的。

保存pc和普通寄存器的值，用于从trap中返回时恢复程序代码的执行。

处理trap时，会进入内核，mode变为supervisor mode, 因为我们需要获取特权使用特权指令来改写控制寄存器。例如进入内核，需要使用内核页表，write satp为内核页表的起始地址。此外，在内核中执行代码，需要从用户空间的栈切换到内核栈: write sp(写sp寄存器不是特权指令，用户空间也经常用)。

总结一下，supervisor mode的特权有:

  • 读写控制寄存器。satp，stvec、sepc、sscratch、scause、sstatus。
  • 使用PTE_U标志位为0的PTE。当PTE_U标志位为1的时候，表明用户代码可以使用这个页表；如果这个标志位为0，则只有supervisor mode可以使用这个页表。

traps from user space

examples:(for user program)

  • makes a system call(ecall instruction)
  • does something illegal(divide by zero or use an invalid virtual address)
  • device interrupts

The high-level path of a trap from user space is uservec(kernel/trampoline.S) then usertrap(kernel/trap.c);and when returning,usertrapret(kernel/trap.c) and then userret (kernel/trampoline.S).

这部分内容强烈建议阅读源码，结合xv6文档读完源码，会理解的十分透彻。

处于用户空间中时，stvec的值为函数uservec()的地址，发生trap时，CPU硬件完成一部分工作(见上)，跳转到uservec()开始执行。

对于函数uservec()：

此时虽然已经进入kernel(supervisor mode)，但页表还未切换(satp中仍为用户页表起始地址),sscratch的值为TRAPFRAME(用户虚拟地址空间的地址),接下来保存31个普通寄存器的值到进程的trapframe中。(因为现在依旧是用户页表，所以指令的中用的地址是相对TRAPFRAME的用户虚拟地址)。

保存完普通寄存器值后，切换到内核栈(write sp)，切换到内核页表(write satp)，跳转到函数usertrap()(jr)。注意write satp后页表已经为内核页表了，jr这条指令对应的地址必须在内核页表中有映射。这里就体现出trampoline页在内核和用户页表中映射到同一虚拟地址的好处了：页表的切换发生在函数uservec()中，映射到同一处可以让uservec()的代码在页表切换前后都能正确执行。

对于函数usertrap():

此时处于kernel(supervisor mode)且页表为内核页表，此时读写trapframe中的寄存器值便不能使用相对TRAPFRAME的用户虚拟地址了，而是直接使用p->trapframe这个物理地址。这里体现出内核虚拟地址空间中RAM地址范围与物理地址直接映射：内核页表可以把这个物理地址当作内核虚拟地址去进行地址转换，反正转换得到的物理地址是一样的。

对于中断的开关问题：

intr_on()和intr_off()都是针对的device interrupt，device interrupt也是一种trap,也会导致CPU硬件以及trap代码修改控制寄存器的内容，因此控制寄存器的内容还需要使用时，不能开中断，防止控制寄存器的内容被覆盖。

Luckily the RISC-V always disables interrupts when it starts to take a trap, and xv6 doesn’t enable them again until after it sets stvec.

例：进入kernel但还没将stvec设置为kernelvec()地址的窗口期，如果再次发生trap(其实就是中断，因为内核中使用ecall，且内核是安全的，不会有除0和缺页错误)，stvec的值仍为uservec(),这就发生了系统性错误。

  2. 本实验的目的是实现个定时器，定时周期到了调用指定的定时器函数，理所当然我们应该在struct proc中增加一个参数保存距离上次定时器函数的调用过去了多长时间，以便时钟中断时判断是否再次执行定时器函数。

  3. 定时器的定时器函数执行前后，程序的原状态要保持不变，因为定时器函数执行过程中可能改变了一些寄存器的值，因此我们需要保存定时器函数执行前程序的状态。

    参考trap机制，保存状态到trapframe中，但原有的trapframe是保存user space ==> kernel space时user space的状态，定时器中断让我们进入内核，然后返回用户空间执行定时器函数，执行定时器函数后user space的状态被改变，因此我们需要一个额外的alarm_trapframe保存程序中定时器函数执行前的状态。新的alarm_trapframe定义在struct proc结构体中。

  4. 此外Prevent re-entrant calls to the handler----if a handler hasn't returned yet, the kernel shouldn't call it again. test2 tests this.

    我们要避免重调用定时器函数：当定时器函数正在执行时，即使定时器定时周期到了，也不调用定时器函数。因此我们需要保存一个值用于判断当前定时器函数是不是正在执行。
