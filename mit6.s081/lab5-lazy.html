MIT6.S081-Lab5 lazy: Lazy allocation

实验目的

本实验的目的是为xv6增加一个lazy allocation的功能，主要应用在进程申请用户虚拟空间中的堆内存时。主体思路就是进程通过sbrk()申请堆内存时，不分配具体的物理页面，只改下p->sz。当进程运行中访问还未分配的页面对应的虚拟地址时，用户页表中没这个映射，触发page fault，此时再分配具体的物理页面，实现lazy allocation的效果。

Eliminate allocation from sbrk()

Your first task is to delete page allocation from the sbrk(n) system call implementation, which is the function sys_sbrk() in sysproc.c. The sbrk(n) system call grows the process's memory size by n bytes, and then returns the start of the newly allocated region (i.e., the old size). Your new sbrk(n) should just increment the process's size (myproc()->sz) by n and return the old size. It should not allocate memory -- so you should delete the call to growproc() (but you still need to increase the process's size!).

本实验就是在系统调用sys_sbrk()中消除实际的物理页面分配。

考虑到sbrk(n)的参数n可能为负:

当\(n < 0\)时，是减少物理页面，这部分不涉及lazy allocation，我们直接copygrowproc()中的代码, 调用uvmdealloc()释放对应的物理页面即可。

当\(n >= 0\)时，是增加物理页面，这部分涉及lazy allocation，根据前面提到的主题思路，我们直接增加p->sz即可。

比如进程初始化后分配了一个堆内存页面，现在进程有4个页面，各自对应一块PGSIZE的地址范围。

页面虚拟地址范围
page 0[addr, addr + PGSIZE)
page 1[addr + PGSIZE, addr + 2 * PGSIZE)
page 2[addr + 2 * PGSIZE, addr + 3 * PGSIZE)
page 3[addr + 3 * PGSIZE, addr + 4 * PGSIZE)

此时 p->sz = addr + 4 * PGSIZE。

Lazy allocation

Modify the code in trap.c to respond to a page fault from user space by mapping a newly-allocated page of physical memory at the faulting address, and then returning back to user space to let the process continue executing. You should add your code just before the printf call that produced the "usertrap(): ..." message. Modify whatever other xv6 kernel code you need to in order to get echo hi to work.

本实验就是处理访问到lazy allocation页面导致的page fault，处理流程也比较简单，就是在usertrap()加上相应的分配具体物理页面。

我们先来看几个寄存器：

  • stval: 触发page fault的虚拟地址。

  • scause: 触发trap的原因, 保存的是一个整数。

    • 8 ==> 系统调用。
    • 12 ==> 指令执行引起的page fault(指令缺页)。
    • 13 ==> load引起的page fault(数据缺页)。
    • 15 ==> store引起的page fault(数据缺页)。

  • sepc: 触发trap的引起程序计数器值。

p->trapframe->sp: 保存进入trap时进程p的stack pointer值，由寄存器sp拷贝至p->trapframe->sp。

弄懂这些寄存器值的含义，看代码就比较清晰了。

注意一点: 堆空间的虚拟地址范围是在[PGROUNDUP(p->trapframe->sp),p->sz)内的。

    系统调用：

    uservec()==>usertrap()中scause()=8对应的分支==>usertrapret()==>userret()

思考

第三个实验中的read和write系统调用处理部分，可不可以这样：

read或write系统调用的地址参数可能为lazy allocation页面(尚未分配物理页面)的地址，read或write读写这个地址时能不能触发新的page fault,导致: trap from kernel --> kernelvec() ==> kerneltrap()==>...

然后我们在kerneltrap()中进行具体的物理页面分配，像usertrap()一样。

答案是不可以。

我试了一下，是不可以的。read或write系统调用最后走到copyout或copyin，copyout 和 copyin 会完成对传入的虚拟地址参数的检查。So kernel is safe！kernel怎么会访问一些没分配内存的页面而导致page fault的trap呢, kernel is smart, user is stupid。
