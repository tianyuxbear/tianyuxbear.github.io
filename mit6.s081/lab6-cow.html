MIT6.S081-Lab6 cow: Copy-on-write fork

实验目的

Virtual memory provides a level of indirection: the kernel can intercept memory references by marking PTEs invalid or read-only, leading to page faults, and can change what addresses mean by modifying PTEs. There is a saying in computer systems that any systems problem can be solved with a level of indirection. The lazy allocation lab provided one example. This lab explores another example: copy-on write fork.

本实验是对xv6中fork的改进，原xv6中fork时子进程拷贝父进程的所有物理页面，现在改进为copy-on-write：对于一个页面，只有父子进程中有写该页面的操作时才复制，否则共享。

主体思路还是利用页表这层抽象，fork时子进程拷贝父进程的页表项但不拷贝页表项对应的物理页面，并且父子进程都把页表项中的PTE_W擦除，父子进程只读取页面内容时，相安无事，但只要有一个进程有write页面的操作，便会触发page fault进入trap，我们在usertrap()完成相应的copy，然后对父子进程中相应页面均添加上PTE_W, 之后父子进程各自有对应的页面，便不会出现读写冲突了。

这个bug我在写博客时是没有的，因为之前主体代码部分已经修复了这个问题。我回忆了一下，当时的问题在于：

usertrap()以及copyout()函数中refc > 1的分支最后一句，刚开始是直接调用sub_refc()减少引用计数，而不是kfree()。

考虑以下情况：对于共享某一个物理页面的两个进程(假设此页面仅被这两个进程共享)，两个进程先后走进refc>1分支(一前一后，离得很近)，这样两个进程各自分配了一个物理页面，拷贝内容到各自新分配的物理页面上，最后即使原来共享的物理页面被两次调用sub_refc()减少引用计数至0，这个页面也没有被释放掉，就造成了内存泄漏。

并发情况下极其复杂，一定要小心内存的申请和释放，加锁和解锁的位置，尽量多思考下所写代码在并发情况下是否会出问题。
